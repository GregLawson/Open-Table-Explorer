Test Driven Design is designed as an antidote to our over-confidence. We are instructed to write tests first, then fix code to pass tests.

Perhaps due to over-confidence I practice a variant of Test Driven Design.

I write code and tests in parallel

Code and tests are alternative specifications of a program

Code is an operational specification (How).

Tests are primarily a declarative specification consisting of Logical Assertions Pre-conditions and Post-conditions (What)

You should have no confidence in the code until the tests are complete and passing

Regression testing allows detection of new bugs that break old code.
