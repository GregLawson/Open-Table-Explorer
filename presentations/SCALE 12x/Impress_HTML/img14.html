You can rewrite Git history by merging commits with either the git-commit –amend command or the git-rebase command. I frequently use the git-commit –amend command (actually the git-cola equivalent) when the same unit is sequentially edited. This obscures how many tries it took me to get it right, making me look more like a genius.

The git-rebase command allows you to easily merge consecutive commits by prefixing them with either fixup! or squash! then the name of the first commit. This works well as long as the merged commits are consecutive, you are careful about the case and spacing in the message, and the first commit does not have the prefix. Git-rebase gets more painful when you rearrange the order of the commits and generate merge conflicts.

The git-rebase –auto-squash command promises to automate this, but because it reorders commits, merge conflicts are likely.

A somewhat tedious but effective method is to use git-rebase without auto-squash and reorder the commits in small increments; using git-rebase –abort when merge conflicts occur. An automated bisection might be possible.

Whether you should you rewrite Git history by merging commits is subject to trade-offs including obscures latent bugs and limiting cherry picking.
